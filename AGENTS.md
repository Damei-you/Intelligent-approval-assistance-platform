# AGENTS.md

本文件用于约束参与本仓库开发的 AI Agent 和开发者。修改代码前应先阅读本文件；子目录没有额外 `AGENTS.md` 时，本文件对整个仓库生效。

## 1. 项目定位

本项目是“智能审批辅助平台”的单企业演示项目，重点展示完整、清晰、可追溯的工程实现，而不是覆盖真实企业全部合同、财务和法务细节。

核心技术栈：

- 后端：Python、FastAPI、Pydantic、psycopg。
- 文档解析：pypdf、Python 原生文本解码和正则切分。
- AI 工作流：LangChain、LangGraph。
- 数据库：PostgreSQL、pgvector。
- 异步任务：Redis、Celery。
- 前端：Vue 3、Vite。
- 本地运行：Docker Compose。

## 2. 当前项目边界

### 已纳入范围

- 采购合同和销售合同两种合同类型。
- PDF、TXT、JSON 合同与制度依据导入。
- 文件解析后先返回标准 JSON，由用户检查或修改。
- 用户确认后，才保存原始文件并事务写入合同、文档和条款。
- 确认导入成功后，通过 Redis/Celery 异步调用 `text-embedding-v4` 生成 1536 维向量。
- 合同条款和企业制度分块的 PostgreSQL/pgvector 数据结构。
- 预定义风险检查项驱动的 RAG 与 LangGraph 审查设计。
- 固定两级审批：业务审批、法务审批。
- 合同多轮问答、引用记录和工作流可观测性的数据结构。

### 当前不纳入范围

- 多租户、用户、角色和权限体系。
- 对接 OA、ERP、合同系统或其他外部业务系统。
- 扫描版 PDF 的 OCR；PDF 必须包含可提取的文本层。
- 文件解析预览阶段的 Embedding；预览只负责解析并返回待确认 JSON。
- 历史合同、标准模板或外部合同库的相似条款检索。
- 可视化审批流程设计器；审批流程暂时固定为两级。
- 复杂制度版本、生效日期、财务规则和法律规则建模。
- 生产级高可用、灾备、计费和合规认证。

当前已完成的业务代码包括“合同/制度解析预览、确认导入、异步向量化”，以及付款、质保、违约责任和争议解决四项 LangGraph 风险审查与前端报告展示。审批和问答当前仍只有数据结构，不要描述成已经实现。

## 3. 关键业务约束

1. 文件预览接口只解析数据，不得保存原始文件，不得写入 PostgreSQL。
2. 前端必须展示解析后的 JSON，并允许用户在确认前修改。
3. 确认导入时必须校验原文件 SHA-256，确保确认文件和预览文件一致。
4. 合同、文档、条款必须在同一个数据库事务中写入；失败时不能留下部分记录。
5. 同一 `contract_no` 再次导入时创建新的文档修订版本，不重复创建合同主记录。
6. 原始文件只有在确认导入后才保存到 `storage/uploads`。
7. 解析预览阶段禁止调用 Embedding；只有确认导入成功后才能创建异步向量化任务。
8. 向量模型固定为 `text-embedding-v4`、1536 维，API Key 只能从 `api-key` 环境变量读取。
9. 向量化失败不能回滚已经成功导入的合同，应通过任务状态记录错误并允许重试。
10. API 错误响应应保持 `{ "code": "...", "message": "..." }` 结构；FastAPI 请求模型的标准校验错误除外。

## 4. Python 中文注释要求

项目使用者正在学习 Python 技术栈，因此所有新增或修改的 Python 注释、文档字符串和面向开发者的说明必须使用中文。

### 必须添加解释型中文注释的场景

- FastAPI 路由装饰器、请求参数来源以及 `response_model` 的作用。
- Pydantic 模型校验、`Field` 限制、校验器和序列化行为。
- `async`/`await`、线程池，以及为什么同步数据库或解析操作不能直接阻塞事件循环。
- psycopg 连接、事务边界、游标批量写入、`Jsonb` 适配等用法。
- `contextmanager`、生成器、数据类、类型注解等不容易理解的 Python 写法。
- 文件哈希、编码回退、PDF 文本提取和条款正则切分逻辑。
- 异常被转换成 HTTP 状态码的位置。
- LangChain、LangGraph、Celery 等框架的关键接入点。
- 为保证业务一致性而采取的非直观实现，例如“预览不落库”和“确认时重新校验文件”。

示例：

```python
# FastAPI 的 UploadFile 使用临时文件封装上传内容，适合处理 multipart/form-data。
# 这里只读取到大小上限再多 1 字节，用于判断文件是否超限，避免无边界读取。
data = await file.read(settings.max_upload_size + 1)
```

```python
# connection.transaction() 会在代码块正常结束时提交，抛出异常时自动回滚，
# 因而合同、文档和条款不会出现只写入一部分的情况。
with connection.transaction():
    ...
```

### 注释风格

- 注释重点解释“为什么这样做”和“框架在这里做了什么”，不要逐行翻译代码。
- 公共类、公共函数和业务边界函数优先使用简短的中文文档字符串。
- 变量名、函数名、类名、数据库字段和 API 字段继续使用英文，不使用拼音命名。
- 技术术语可保留英文，但第一次出现时应给出简短中文含义，例如“事务（Transaction）”。
- 注释必须与当前实现一致；修改逻辑时同步修改过期注释。
- 不在注释中记录密码、API Key、访问令牌或真实合同数据。

## 5. 代码组织原则

- `app/core`：配置、数据库连接等基础设施。
- `app/modules`：后端业务模块。
- `frontend`：Vue 3 前端。
- `database/init`：首次创建 PostgreSQL 数据卷时执行的初始化 SQL。
- `docs`：数据库和 API 设计文档。
- `tests`：单元测试和集成测试。
- `examples`：可以提交到仓库的脱敏演示数据。
- `storage/uploads`：运行时上传文件，不得提交真实内容。

保持模块规模与功能复杂度相匹配。不要为了展示“架构”新增没有实际用途的抽象层、接口类或文件；已有模块能清晰承载功能时，应优先在已有模块中修改。

## 6. 数据库与迁移

- 主键使用 UUID，时间使用 `TIMESTAMPTZ`，动态扩展数据使用 `JSONB`。
- 向量统一存放在 `document_chunks.embedding`。
- `database/init` 只会在空数据卷首次启动时执行。已有数据库结构发生变化时，应增加迁移方案，不能假设重启容器会重新执行初始化脚本。
- 业务表写入应使用参数化 SQL，禁止拼接用户输入。
- 删除或覆盖数据前必须明确目标，测试只允许清理测试自身创建的数据。

## 7. 安全与配置

- 不得提交 `.env`、真实密码、API Key、Token、私钥或真实合同文件。
- `.env.example` 只提供演示配置和占位值。
- `compose.yaml` 中的 `L123456` 仅为本地演示默认密码，不得用于正式环境。
- 日志和异常响应不得输出数据库密码、模型密钥或完整敏感合同内容。
- 上传文件必须校验扩展名、大小和解析结果；保存文件名必须进行安全处理。

## 8. 验证要求

修改 Python 后端后至少执行：

```powershell
.\.venv\Scripts\python.exe -m compileall -q app tests main.py
$env:RUN_INTEGRATION_TESTS='1'
.\.venv\Scripts\python.exe -m unittest discover -s tests -v
```

修改 Vue 前端后至少执行：

```powershell
Set-Location frontend
npm run build
```

接口、请求字段或业务流程改变时，必须同步更新：

- `docs/contract-import-api.md`
- `api-tests.http`
- 相关自动化测试
