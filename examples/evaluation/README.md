# 风险审查与重排序评测数据集

本目录提供完全虚构的合同与制度数据，仅用于验证付款、质保、违约责任和争议解决四项风险审查，不代表真实企业制度或法律意见。

## 数据组成

- `policies/`：同一制度编号的新旧两个版本。必须先导入 V1.0，再导入 V2.0，确保 V2.0 为当前版本。
- `contracts/`：12 组可直接提交到合同 JSON 导入接口的测试合同。
- `expected-results.json`：每组合同的预期状态和应命中的主要证据。
- `rerank-judgements.json`：困难样本的分级相关性标注，`2` 表示直接相关，`1` 表示部分相关，`0` 表示不相关或具有误导性。

## 建议执行顺序

1. 导入 `policy-v1-old.json` 并等待向量化完成。
2. 导入 `policy-v2-current.json` 并等待向量化完成；旧版本应自动变为非当前版本。
3. 逐个导入合同并等待向量化完成。
4. 对每份合同发起风险审查，对照 `expected-results.json` 检查状态和证据。
5. 引入重排序后，保持查询、候选集合和最终 Top K 不变，对照 `rerank-judgements.json` 比较排序指标。

## 两层评测

第一层只检查检索，不调用聊天模型：

- 合同证据 `Recall@20`、`MRR`、`Precision@3`、`NDCG@3`。
- 制度证据 `Recall@20`、`MRR`、`Precision@5`、`NDCG@5`。
- 跨合同污染必须为 0，旧制度引用必须为 0。

第二层检查完整 RAG 结果：

- `PASS/RISK/INSUFFICIENT_INFORMATION` 状态准确率。
- 非信息不足结论的合同与制度双侧证据完整率。
- 引用合法率和错误引用困难负样本的次数。

重排序对比时，建议向量阶段统一召回 Top 20：基线直接截取前 3/5，实验组对同一批 Top 20 重排序后再截取前 3/5。重排序不能补救召回阶段完全漏掉的证据。
